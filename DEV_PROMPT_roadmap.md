# Промт для разработки: FeynMap — доработки (Roadmap)

> Реализуемая постановка доработок поверх базового FeynMap и корпоративного слоя. Контекст и обоснование — в `ROADMAP.md`. Базовые ТЗ — `DEV_PROMPT.md` (MVP) и `DEV_PROMPT_corporate.md` (роли, БЗ, RAG, админка). Код — в репозитории проекта.

Сделай по вехам M1→M5. Внутри вехи задачи можно параллелить. Каждая задача — с критерием приёмки.

---

## Сквозные требования (применять везде)

- **Provider-agnostic слой.** Введи интерфейсы `LLMProvider` (chat/complete) и `EmbeddingProvider` (embed) с реализациями `anthropic`, `openai`, `local` (vLLM/OpenAI-совместимый эндпоинт). Выбор провайдера — через ENV/конфиг, без правок бизнес-логики. Эндпоинты `/chat`, `/assess`, `/report` и RAG-ретрив ходят только через эти интерфейсы.
- **Языки: RU / UZ-Latn / EN.** Узбекская кириллица не поддерживается — нормализацию Latn↔Cyrl не делать.
- **Security-by-design.** Права проверяются на сервере; ничего секретного в браузер; данные изолированы по `org_id`.
- **Human-in-the-loop** для оценок с последствиями.

---

## M1 — Фундамент

### 1.1 Слой-абстракция LLM и эмбеддингов
- Интерфейсы + реализации (`anthropic`/`openai`/`local`), фабрика по конфигу, таймауты/ретраи, единый формат ошибок.
- **Критерий:** переключение провайдера для Студента/Ассессора/эмбеддингов — только сменой ENV; код не меняется; есть smoke-тест на каждый провайдер.

### 1.2 i18n RU / UZ-Latn / EN
- Фронт: словари локалей, переключатель языка, локаль в профиле пользователя/сессии.
- Бэк: системные промпты Студента/Ассессора/Report параметризованы языком; язык передаётся в сессию.
- **Критерий:** вся сессия (вопросы Студента, отчёт, UI) идёт на выбранном языке; узбекский — только латиница.

### 1.3 Базовый security-чек-лист
- Секреты только на бэке и в секрет-менеджере; `.env` не в репозитории.
- TLS; CORS-allowlist; заголовки безопасности (helmet); валидация входов.
- RBAC-middleware покрывает все новые эндпоинты; добавить тесты «роль X не может Y».
- Аудит-лог расширить: загрузка БЗ, назначения, **override оценок**, доступ к результатам.
- **Критерий:** автотесты RBAC зелёные; секрет-скан в CI; в логах нет ключей/ПДн.

## M2 — Доверие к оценке

### 2.1 Прозрачный рубрик оценки
- Ассессор возвращает по каждой компетенции не только `score/evidence/gap`, но и **критерий** (почему такой балл относительно эталона БЗ) со ссылкой на источник (`source_refs`).
- В отчёте и в ревью — показать рубрик и цитату эталона.
- **Критерий:** для любого балла видно «за что» и ссылку на пункт материала.

### 2.2 Human-in-the-loop и апелляция
- Экран ревью Ассессора (из `DEV_PROMPT_corporate.md`) дополнить: корректировка балла с обязательным комментарием → `overridden=true`, запись в аудит.
- Обучающийся может подать апелляцию по результату (статус: подана/рассмотрена).
- **Критерий:** оценка с последствиями не финализируется без подтверждения Ассессора; апелляция трекается.

### 2.3 Антифрод
- Детекция вставки больших фрагментов в поле объяснения (paste-событие, аномальный темп ввода, резкая смена стиля).
- Флаг подозрительной сессии в результате + в дашборде Ассессора.
- **Критерий:** массовая вставка текста из внешнего ассистента помечается флагом; флаг виден ревьюеру.

## M2.5 — Психологическая безопасность / Wellbeing

> Поднято инвестором на питче («как компенсируете стресс?»). Цель — не создавать стресс ассессмента by design. Слой проектировать с участием психолога. Тезисы — в `PITCH_QA.md`.

### 2.5.1 Разделение режимов «Практика» и «Ассессмент»
- **Практика:** приватна (результаты по умолчанию не уходят HR), без ярлыков-оценок (карта роста вместо «двойки»), свободные переигровки, без дедлайнов. Режим по умолчанию.
- **Ассессмент:** запускается только по явному согласию/назначению, с понятными ставками и человеческим ревью (M2.2).
- **Критерий:** пользователь всегда видит текущий режим и кто увидит результат; практика не попадает в HR-аналитику без явного перевода в ассессмент.

### 2.5.2 Поддерживающий тон и подача результатов
- Системный промпт Студента: тёплый, поощряющий, отмечает усилие и микро-победы, без сарказма; уровень эмпатии — настраиваемый параметр.
- Отчёт в логике роста: сначала сильные стороны, пробелы названы «зонами роста», язык growth-mindset, акцент на динамике во времени.
- **Критерий:** в отчёте сильные стороны идут первыми; формулировки пробелов проходят ревью на отсутствие осуждающего тона.

### 2.5.3 Адаптация под эмоциональное состояние
- Сигналы фрустрации: аномальный темп ввода, резкие удаления, негативный сентимент текста; в голосовом режиме (M3.1) — тон.
- Реакция: Студент смягчается, предлагает подсказку, паузу или завершение **без штрафа**; никаких таймеров и публичных рейтингов по умолчанию.
- Скаффолдинг при буксовке вместо «провала».
- **Критерий:** при устойчивых сигналах фрустрации система предлагает паузу/подсказку; выход в любой момент не ухудшает результат.

### 2.5.4 Тревога → действие
- По пробелам — конкретный план «что изучить» (связь с циклом доучивания M3.3) и право на повтор (цель — мастерство, не «поймать»).
- Прозрачный рубрик (M2.1) и апелляция (M2.2) как опора доверия.
- **Критерий:** после сессии пользователь видит план и может перепройти; повтор не штрафуется.

### 2.5.5 Онбординг-рамка и настройки wellbeing
- Стартовый экран задаёт ожидания: «безопасная тренировка, ошибаться — норма и смысл».
- Настройки тона/режима валидируются психологом; опционально — агрегированный (неперсональный) «индекс нагрузки» для HR (не наказательный).
- **Критерий:** первый вход показывает рамку безопасности; HR-индекс нагрузки — только агрегат, без привязки к конкретному человеку.

## M3 — Дифференциаторы

### 3.1 Голосовой режим
- Ввод объяснения голосом (STT) → тот же конвейер оценки; вопросы Студента озвучиваются (TTS).
- STT/TTS — за провайдер-абстракцией (облачный сейчас, локальный позже); поддержка RU/UZ-Latn/EN.
- **Критерий:** можно пройти всю сессию голосом; транскрипт сохраняется в историю.

### 3.2 Leveling (цели и уровни)
- Профиль компетенций под должность/роль (целевые уровни по компетенциям).
- В отчёте — gap-to-target, а не только общий балл.
- **Критерий:** назначение можно привязать к профилю должности; отчёт показывает разрыв до цели.

### 3.3 Цикл доучивания
- По выявленным пробелам — рекомендовать конкретные разделы/чанки БЗ; кнопка «переэкзамен»; расписание повторений (spaced repetition).
- **Критерий:** после сессии обучающийся видит, что изучить, и может перепройти; повторы планируются.

### 3.4 Уведомления
- Дедлайны и напоминания: email + Telegram-бот.
- **Критерий:** при назначении и перед дедлайном приходит уведомление по выбранному каналу.

## M4 — Локализация и комплаенс

### 4.1 On-prem LLM
- Развернуть локальную модель (старт — Qwen 32B–72B) через vLLM (OpenAI-совместимый эндпоинт); подключить как провайдер `local`.
- Эмбеддинги — BGE-m3 локально.
- **Критерий:** полный цикл (Студент + Ассессор + RAG) работает на локальном стеке через смену конфига.

### 4.2 Eval на узбекском
- Набор 30–50 тем UZ-Latn с эталонными объяснениями; метрики: качество вопросов, точность оценки против источника, валидность строгого JSON.
- Сравнить облачный baseline vs Qwen (базовый / + LoRA).
- **Критерий:** есть отчёт сравнения; принято решение go/no-go по локальной модели с цифрами.

### 4.3 Локализация ПДн (РУз)
- Перенос хранения/обработки ПДн граждан РУз на серверы в стране; шифрование at-rest; политика хранения/удаления.
- **Критерий:** прод-данные реальных сотрудников не покидают РУз; есть политика ретеншена.

## M5 — Интеграции

- **SSO/SAML/OIDC** — корпоративный вход; MFA для админов.
- **LMS (SCORM/xAPI) и HRIS** — экспорт результатов/прогресса.
- **Версионирование БЗ** — фиксировать версию документа, использованную в сессии; при переиндексации не ломать прошлые результаты.
- **Когортная аналитика** — сравнение групп, тренды во времени, экспорт.
- **Критерий:** вход через корпоративный IdP; результаты уходят в LMS/HRIS; в карточке сессии видна версия источника.

---

## Порядок и зависимости

- M1 — предпосылка для всего (абстракция, i18n, security).
- M2 + M2.5 — до запуска ассессмента с последствиями (доверие к оценке + психологическая безопасность идут в паре).
- M4 зависит от M1 (абстракция) и проверяется eval’ом до переключения.
- M3 и M5 — параллельно после M1/M2. M2.5.3 (детекция фрустрации по тону) частично опирается на голосовой режим M3.1.
