# Промт для разработки: FeynMap — тренажёр «учусь, объясняя» + живая карта компетенций

> Этот документ — самодостаточное задание на разработку. Передаётся разработчику (или ИИ-агенту-кодеру) вместе с `architecture.drawio`. Цель — за 2 дня хакатона собрать работающее, эффектное на сцене приложение. Один сценарий, доведённый до полировки, важнее множества сырых фич.

---

## 1. Что строим

Приложение объединяет две механики в одну историю:

- **Feynman-режим (обучение через объяснение).** Пользователь выбирает/вводит тему и объясняет её ИИ-«наивному студенту». Студент задаёт уточняющие/наивные вопросы и вскрывает пробелы в понимании.
- **Адаптивный ассессмент с живой картой.** Параллельно вторая ИИ-роль («Ассессор») оценивает продемонстрированные компетенции по ходу диалога и в реальном времени заполняет радар-карту навыков. Следующие вопросы Студента адаптивно бьют в слабые/незакрытые зоны.
- **Итог.** Финальный экран: карта компетенций, список пробелов и 3–5 коротких рекомендаций, что подтянуть.

**Ценность:** для edtech — диагностика и обучение через объяснение; для HRTech — оценка реального уровня владения темой (а не заявленного в резюме).

Архитектура простая: один цикл «промпт → LLM → структурированный ответ → UI». Без RAG, без файнтюна, без своей БД.

---

## 2. Стек (держим простым)

- **Frontend:** React + Vite + TypeScript + Tailwind. Чат-панель слева, живой радар (recharts `RadarChart`) справа, отдельный экран финального отчёта.
- **Backend:** минимальный Node + Express как прокси к LLM. **API-ключ только на бэке, никогда в браузере.** Эндпоинты `/chat` и `/assess` (+ опционально `/report`).
- **LLM:** Anthropic Claude. По умолчанию `claude-sonnet-4-6`. Для роли «Ассессор» — `claude-haiku-4-5` ради скорости и цены.
- **Состояние:** React state + `localStorage` для текущей сессии. БД не нужна.
- **Никаких HTML `<form>`** — сабмиты через `onClick` / `onChange` / `onKeyDown`.

## 3. Архитектура

Полная схема потока данных и диаграмма последовательности одного хода — в файле **`architecture.drawio`** (две вкладки: «Архитектура — поток данных» и «Адаптивный цикл одного хода»).

Кратко поток:
```
Пользователь → Frontend (Чат-панель) → /chat → Студент → вопрос обратно во Frontend
                       └→ история → /assess → Ассессор → JSON-оценка → Радар
                                                        └→ next_focus → Студент
```

Порядок на каждый ход пользователя: **сначала `/assess`** (обновить радар + получить `next_focus`), **затем `/chat`** с этим `next_focus` (адаптивный вопрос).

## 4. Две роли LLM (через системные промпты)

### 4.1. Студент (виден пользователю)
Наивный, любопытный, темы не знает. На вход: тема + история диалога + (опц.) подсказка `next_focus`.

**Поведение:**
- Ровно ОДИН короткий вопрос за ход (1–3 предложения).
- Цепляется за неясности и жаргон, просит примеры и аналогии, спрашивает «почему / а что если».
- Не читает лекций, не даёт ответов сам, не выдаёт, что знает тему.
- Если пришёл `next_focus` — мягко направляет вопрос в эту слабую зону, но не цитирует подсказку.
- Вывод — только текст вопроса, без префиксов и markdown.

### 4.2. Ассессор (невидим пользователю) — СТРОГО JSON
На вход: тема + история диалога (+ опц. предыдущая оценка для стабильности состава).

**Поведение:**
- На первом ходу формирует набор из 5–7 компетенций под тему; далее НЕ меняет состав без необходимости, только обновляет `score`/`evidence`/`gap`.
- Оценивает РЕАЛЬНО продемонстрированное, а не заявленное.
- `score` 0–100; `gap=true`, если зона провалена/не раскрыта (score < 40).
- `next_focus` — имя одной компетенции с самым важным незакрытым пробелом.

**Выход — строго валидный JSON, без markdown и преамбулы:**
```json
{
  "competencies": [
    {"name": "string", "score": 0, "evidence": "string", "gap": false}
  ],
  "next_focus": "string",
  "overall": 0
}
```

**Парсинг безопасный:** срезать ```` ```json ````-фенсы, взять подстроку от первого `{` до последнего `}`, `try/catch`. При ошибке парсинга — не ронять UI: вернуть предыдущую оценку либо пустую `{competencies: [], next_focus: "", overall: 0}`.

### 4.3. (Опц.) Рекомендации для финала
Отдельный мелкий вызов (можно на `haiku`): на вход тема + итоговая оценка, на выход `{ "recommendations": ["string", ...] }` — 3–5 коротких конкретных советов, опираясь на зоны с низким score и `gap=true`.

## 5. API-контракты (бэкенд)

Все эндпоинты — `POST`, тело и ответ JSON. Прокси пробрасывает на Anthropic, ключ — из `.env`.

| Эндпоинт | Запрос | Ответ |
|---|---|---|
| `/chat` | `{ topic: string, history: Msg[], next_focus?: string }` | `{ question: string }` |
| `/assess` | `{ topic: string, history: Msg[], previous?: Assessment \| null }` | `Assessment` (см. JSON выше) |
| `/report` | `{ topic: string, assessment: Assessment }` | `{ recommendations: string[] }` |

`Msg = { role: 'student' | 'user', content: string }`. На стороне Claude: `student → assistant`, `user → user`. Диалог для `/chat` должен начинаться с реплики пользователя (при пустой истории — подставить служебную user-реплику «начни диалог»). Для `/assess` удобнее склеить историю в один текстовый транскрипт и передать как одно `user`-сообщение.

Нормализовать выход Ассессора: `score`/`overall` зажать в [0, 100], `gap` привести к boolean.

## 6. Frontend: экраны и логика

**Три экрана** (`screen: 'topic' | 'chat' | 'report'`):

1. **Экран темы.** Поле ввода темы + 4–5 примеров-чипов. Кнопка «Начать». (Опц.) «Продолжить прошлую сессию» из `localStorage`.
2. **Экран диалога.** Две колонки: слева чат-панель (история + поле ввода, индикатор «студент думает»), справа живой радар + список компетенций с прогресс-барами и общим баллом. Кнопки «Новая тема» и «Завершить и собрать отчёт».
3. **Экран отчёта.** Итоговый радар + общий балл, сильные стороны, пробелы (`gap=true`) с evidence, блок «Что подтянуть» (рекомендации). Кнопки «Продолжить объяснять» / «Новая тема».

**Логика хода `send(text)`:**
1. Добавить `{role:'user', content:text}` в историю; показать «студент думает».
2. Вызвать `/assess` → обновить `assessment` (радар оживает с анимацией) → запомнить `next_focus`. Ошибка оценки не критична — не блокирует диалог.
3. Вызвать `/chat` c `next_focus` → добавить `{role:'student', content:question}`.

**Старт сессии:** при выборе темы вызвать `/chat` с пустой историей → получить приветственный вопрос Студента.

**Состояние и хранение:** `topic`, `history`, `assessment` в React state; зеркалить в `localStorage` (ключ `feynmap-session`). Радар обязан перерисовываться по мере обновления оценки (recharts `isAnimationActive`).

## 8. Порядок сборки (по фазам — не усложнять раньше времени)

1. Скаффолд репозитория (frontend + backend), `.env.example` с `ANTHROPIC_API_KEY`, README.
2. Рабочий чат-цикл с ролью «Студент» (тема → диалог).
3. Роль «Ассессор» + безопасный парсинг JSON.
4. Радар-карта, привязанная к выходу Ассессора, обновляется в реальном времени по ходу диалога.
5. Адаптивность: `next_focus` от Ассессора передаётся Студенту как подсказка для следующего вопроса.
6. Финальный экран: итоговая карта + пробелы (`gap=true`) + 3–5 рекомендаций.
7. Полировка демо: одна заготовленная тема-пример, плавная анимация заполнения радара, аккуратная типографика.

## 10. Критерии готовности (для демо на сцене)

- [ ] Пользователь вводит тему → Студент здоровается и задаёт первый вопрос.

- [ ] Ключ Anthropic нигде не утекает в браузер (только бэк).
