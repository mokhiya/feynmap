# FeynMap — Roadmap доработок

Стратегический план развития поверх базового MVP и корпоративного слоя (см. `DEV_PROMPT.md`, `DEV_PROMPT_corporate.md`). Реализуемая постановка для коллеги — в `DEV_PROMPT_roadmap.md`.

## 0. Сквозные принципы

- **Provider-agnostic с первого дня.** LLM и эмбеддинги — за интерфейсом-абстракцией, смена облако↔локально = смена конфига, не переписывание.
- **Security-by-design.** Права на сервере, изоляция по организации, минимизация ПДн.
- **Языки: RU / UZ-Latn / EN** (узбекская кириллица не поддерживается).
- **Human-in-the-loop** для оценок с последствиями.
- **Психологическая безопасность.** Продукт спроектирован не создавать стресс ассессмента: ролевой переворот (пользователь — учитель, не экзаменуемый), безопасный режим практики, развитие вместо «трибунала». См. M2.5 и `PITCH_QA.md`.

## 1. Функциональные доработки

### P0 — без этого продукт неполон для рынка
1. **Мультиязычность RU / UZ-Latn / EN** — UI (i18n), контент базы знаний и сам диалог. Без нормализации Latn↔Cyrl (кириллица убрана).
2. **Калибровка и прозрачность оценки** — видимый рубрик (по каким критериям балл), human-in-the-loop, апелляция. Оценка должна быть юридически защитима.
3. **Антифрод для режима ассессмента** — детекция вставки текста (paste из ChatGPT), анализ темпа/стиля, целостность сессии.

### Психологическая безопасность / Wellbeing (сквозной слой — поднят на инвест-питче)
Вопрос со сцены: «как компенсируете стресс?». Ответ — не создавать его by design (см. `PITCH_QA.md`). В софт:
- Разделение режимов **Практика** (приватно, без ярлыков-оценок, свободные переигровки) и **Ассессмент** (по согласию, ревью человека); по умолчанию — практика.
- Поддерживающий тон Студента, результаты в логике роста (сначала сильные стороны, пробелы = «зоны роста»).
- Адаптация под состояние: детекция фрустрации → смягчение/подсказки/пауза; право выйти без штрафа; нет таймеров и публичных рейтингов по умолчанию.
- Тревога → действие: план «что изучить» + право на повтор; прозрачный рубрик и апелляция.
- Онбординг-рамка: «безопасная тренировка, ошибаться — норма». Слой проектировать **с участием психолога**.

### P1 — сильные дифференциаторы
4. **Голосовой режим** — объяснение вслух (STT → оценка), вопросы Студента голосом (TTS). Метод Фейнмана вживую; опирается на экспертизу Spitch.
5. **Цели и уровни (leveling)** — профиль компетенций под должность, gap-to-target вместо «общего балла».
6. **Цикл доучивания** — после пробелов рекомендовать разделы БЗ → переэкзамен → spaced repetition.
7. **Уведомления** — дедлайны и напоминания (email + Telegram).

### P2 — интеграции и зрелость
8. SSO/SAML/OIDC.
9. Интеграции LMS (SCORM/xAPI) и HRIS.
10. Версионирование документов БЗ (какая версия источника использовалась в сессии).
11. Когортная аналитика и тренды во времени.
12. SME-ревью ИИ-сгенерированных тем/компетенций.
13. PWA / мобильный режим.

## 2. Безопасность

### Юридический P0 (специфика РУз)
- **Закон РУз «О персональных данных»** требует хранения и обработки ПДн граждан Узбекистана на серверах **в РУз**. Результаты ассессмента = чувствительные ПДн.
  - Облачное тестирование — **только синтетические/обезличенные данные**.
  - Прод с реальными сотрудниками — в стране (см. раздел 3: переход на локальный стек).

### Технический чек-лист
- **Секреты** — ключи только на бэке, в секрет-менеджере (SysPass), не в репозитории.
- **AuthZ** — RBAC на сервере, least privilege, scope по `org_id`, MFA для админов.
- **Изоляция арендаторов** — row-level security по организации.
- **Шифрование** — TLS в транзите; at-rest для БД и файлов; чувствительные поля результатов отдельно.
- **Prompt injection** — документы БЗ и реплики пользователя не могут переопределять системные инструкции; защита Ассессора от «накрутки балла» через jailbreak.
- **Загрузка файлов** — антивирус-хук, лимиты типа/размера, без path traversal.
- **Vendor data usage** — при облаке только zero-retention / enterprise-условия (no training on data).
- **Аудит** — кто менял баллы (override), кто смотрел результаты; tamper-evident лог.
- **Rate limiting и бюджеты токенов** — против абьюза и расходов.
- **Бэкапы и DR**, контроль зависимостей (supply-chain), пентест перед продом.

## 3. Облако → локальный LLM (план миграции)

1. **Слой-абстракция** LLM и эмбеддингов — заложить сразу.
2. **Фаза «Облако»** (Anthropic Claude + OpenAI embeddings) — скорость итераций, лучшее качество, валидация **методики оценки**. Данные — синтетические/обезличенные.
3. **Фаза «Локально/on-prem»** — соответствие закону о ПДн, конфиденциальная БЗ, стоимость на масштабе, независимость от вендора.
4. **Возможен гибрид** — тяжёлое рассуждение/оценку на лучшей доступной модели, эмбеддинги/перевод/работу с ПДн локально. Но всё, что касается ПДн граждан РУз — в стране.
5. **Переключение — только после бенчмарка** на собственном узбекском наборе (см. раздел 4).

## 4. Локальный LLM для узбекского (выбор + eval)

> Узбекский (только латиница) — низкоресурсный язык. Оценки даны на начало 2026 — **обязателен бенчмарк на собственном наборе** перед выбором.

### Генеративная модель — старт с Qwen (Alibaba)
- Qwen2.5 / Qwen3 (32B–72B): обычно лучшее открытое мультиязычное качество, приличный узбекский/тюркские, хорошее следование инструкциям, рабочая лицензия. Разворачивание через vLLM на on-prem GPU.
- Для сравнения: Gemma 2/3, Llama 3.3 (узбекский слабее).
- **Aya Expanse** — мультиязычная, но CC-BY-NC (некоммерческая) → для корпората не подходит.
- Для узбекского, вероятно, понадобится **LoRA-дообучение** на доменных данных RU/UZ-Latn/EN. Источник данных/моделей — узбекские NLP-наработки (напр., Tahrirchi).

### Эмбеддинги для RAG — BGE-m3 (BAAI)
- Отличное мультиязычное покрытие (вкл. низкоресурсные), dense+sparse, локально — снимает стоп-фактор отправки документов наружу.
- Альтернатива: multilingual-e5-large.

### Eval-план
- Набор 30–50 узбекских (UZ-Latn) тем с эталонными объяснениями.
- Метрики: качество наводящих вопросов Студента; точность оценки Ассессора против источника; стабильность строгого JSON.
- Сравнить: облачный baseline vs кандидаты (Qwen базовый / Qwen+LoRA).

### Железо
- 32–72B → GPU (H100/A100 или 2×A6000). Влияет на TCO on-prem.
