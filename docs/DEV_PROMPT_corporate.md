# Промт для разработки: FeynMap Corporate — база знаний, роли, назначение тем и уроков

> Это **надстройка над базовым FeynMap** (см. `DEV_PROMPT.md` и `architecture.drawio`). Ядро — диалог «Студент ↔ объясняющий» + «Ассессор» + живая радар-карта компетенций — **сохраняем**. Здесь добавляем корпоративный слой: загрузку базы знаний, роли с правами, темы/уроки и назначения. Позиционирование: **корпоративное обучение и ассессмент**.

---

## 1. Что добавляем (суть)

Компания загружает **базу знаний** (учебник по тестированию, PMBoK, курсы по Python и т.д.). На её основе формируются **темы** и **уроки**. Дальше — две модели работы обучающегося:
- **Свободный обучающийся** — сам выбирает тему из доступного каталога.
- **Назначаемый сотрудник** — выполняет темы/уроки, которые ему назначил HR / ассессор / наставник.

После выбора/назначения запускается «магия» базового FeynMap — **но теперь grounding-ом служит загруженная база знаний** (RAG-эталон): Студент задаёт вопросы по материалу, а Ассессор оценивает объяснение **на соответствие первоисточнику**, а не «в вакууме».

**Ключевая ценность для корпората:** объективная оценка реального владения темой относительно эталонного материала + управляемое назначение и аналитика по людям/командам.

---

## 2. Изменения в стеке (относительно базового MVP)

Базовый FeynMap был «без БД и без RAG». Корпоративный слой это снимает:

- **БД:** PostgreSQL 16 + расширение **pgvector** (хранение эмбеддингов чанков базы знаний).
- **ORM:** Drizzle (TypeScript) — или Prisma на усмотрение команды.
- **Авторизация:** JWT + bcrypt; RBAC-middleware на бэке (права проверяются **на сервере**, не только в UI).
- **Парсинг документов:** PDF (`pdf-parse`), DOCX (`mammoth`), TXT/MD — нативно.
- **Эмбеддинги:** OpenAI `text-embedding-3-small` (ключ — только на бэке). Чат/оценка — Anthropic Claude (как в базе).
- **Индексация БЗ:** очередь (BullMQ + Redis) либо инлайн-обработка для MVP.
- **Хранение файлов:** S3-совместимое хранилище или локальный диск для MVP.
- Frontend — тот же React + Vite + TS + Tailwind, добавляются ролевые интерфейсы.

## 3. Ролевая модель (дробная) и матрица прав

Шесть ролей. Права проверяются на бэке для каждого эндпоинта; UI скрывает недоступное.

| Роль | Назначение |
|---|---|
| **Администратор** | Управляет организацией, пользователями, ролями и **базой знаний** (загрузка/курирование документов). |
| **HR** | Назначает темы/уроки в масштабе организации, видит аналитику по всей компании. Не курирует контент БЗ и не задаёт критерии оценки. |
| **Ассессор (Эксперт)** | Определяет компетенции и критерии оценки по теме, **проверяет и при необходимости корректирует** авто-оценку, назначает ассессмент. |
| **Наставник** | Работает со своей группой подопечных: назначает уроки/темы, смотрит их прогресс, даёт обратную связь. |
| **Обучающийся — свободный** | Сам выбирает тему из каталога **и** выполняет назначенное. |
| **Обучающийся — назначаемый** | Выполняет **только назначенное**, без свободного выбора темы. |

## 4. Модель данных (ключевые сущности)

- **Organization** — `id, name, settings`. (Мультиарендность заложить, даже если запуск на одну организацию.)
- **User** — `id, org_id, name, email, password_hash, role(s), mentor_id?, status`.
- **Role / Permission** — роль = набор permissions (см. матрицу).
- **Document** (база знаний) — `id, org_id, title, source_type(pdf/docx/txt/md), file_ref, status(uploaded/parsing/indexed/failed), uploaded_by, created_at`.
- **DocumentChunk** — `id, document_id, text, embedding vector, metadata(page/section/heading)`. Индекс pgvector (ivfflat/hnsw).
- **Topic** — `id, org_id, title, description, status(draft/published), created_by`. Связь **Topic ↔ Document(s)** (какой материал БЗ заземляет тему) + опц. ссылки на конкретные чанки/разделы.
- **Competency** — `id, topic_id, name, description, weight`. Формируется Ассессором или предлагается ИИ из материала, затем утверждается.
- **Lesson** — `id, org_id, title, description, ordered_topic_ids[]`. Учебный блок = упорядоченный набор тем + интро.
- **Assignment** — `id, assignee_id, assigner_id, target_type(topic/lesson), target_id, due_at, status(assigned/in_progress/done/overdue), created_at`.
- **Session** — `id, learner_id, topic_id, assignment_id?, history(jsonb), started_at, completed_at`. (Это и есть прохождение базовой «магии».)
- **AssessmentResult** — `id, session_id, competencies(jsonb: name/score/evidence/gap), overall, next_focus, recommendations(jsonb), source_refs(jsonb: какие чанки/разделы БЗ использованы как эталон), reviewed_by?, overridden(bool)`.
- **AuditLog** — кто/что/когда (загрузки БЗ, назначения, override оценок).

## 5. RAG-конвейер (как БЗ заземляет «магию»)

**Индексация (при загрузке документа):**
1. Загрузка файла → валидация типа/размера.
2. Парсинг в текст (с сохранением структуры: страницы/заголовки).
3. Чанкинг (~600–900 токенов, overlap ~15%), сохранение `DocumentChunk` с metadata.
4. Эмбеддинги (`text-embedding-3-small`) → запись вектора в pgvector.
5. Статус документа → `indexed`. Прогресс виден в UI Админа.

**Во время сессии:**
- При старте по `topic_id` достаём связанные документы темы.
- На каждый ход: ретрив релевантных чанков (по теме + текущей реплике обучающегося, top-k) → инъекция в системные промпты как блок **«Справочный материал (эталон)»**.
- **Студент** (sonnet): задаёт вопросы, опираясь на материал; вскрывает то, что в эталоне есть, а обучающийся не сказал.
- **Ассессор** (haiku): оценивает объяснение **на соответствие эталону** — `score` отражает и глубину, и фактическую верность источнику; `evidence` ссылается на пункт материала; в `source_refs` сохраняем использованные чанки. Если объяснение противоречит эталону — это явный gap.

> Эндпоинты `/chat`, `/assess`, `/report` базового FeynMap расширяются: принимают `topic_id`/`session_id`, на бэке подмешивают retrieved-контекст. Контракт ответа (вопрос / строгий JSON оценки / рекомендации) — без изменений.

## 6. Функциональность и экраны по ролям

### Администратор — админ-панель
- **Пользователи (заведение и управление):**
  - создание пользователя вручную (имя, email, роль, для наставника — привязка подопечных) с выдачей пароля или ссылки-приглашения на email;
  - **массовый импорт** пользователей из CSV/XLSX (для онбординга компании);
  - редактирование пользователя, смена роли, привязка к наставнику/команде;
  - блокировка/разблокировка и деактивация (без удаления истории), сброс пароля;
  - поиск и фильтр по роли/статусу/команде, пагинация;
  - все действия фиксируются в `AuditLog`.
- **Роли и права:** просмотр 6 преднастроенных ролей и их permissions; назначение роли пользователю. (Редактор кастомных ролей — отдельной фазой, на вырост.)
- **База знаний:** загрузка документов (drag-and-drop), список со статусами индексации, переиндексация, удаление, просмотр исходника.
- **Настройки организации:** название, домены для приглашений, дефолтные параметры сессий.

### Назначающий (HR / Ассессор / Наставник)
- **Конструктор тем и уроков:** выбрать документ(ы) БЗ → кнопка «Предложить темы и компетенции из материала» (ИИ-черновик) → редактирование/утверждение → публикация. Сбор тем в **урок** (упорядоченный список).
- **Ассессор дополнительно:** редактор компетенций и критериев, экран **ревью авто-оценки** (посмотреть результат, при необходимости скорректировать score/комментарий → `overridden=true`).
- **Консоль назначений:** выбрать обучающегося/группу → выбрать тему или урок → дедлайн → назначить. Таблица назначений со статусами (назначено / в работе / готово / просрочено).
- **Аналитика:** карта компетенций по команде/организации (агрегированный радар + heatmap пробелов), индивидуальные отчёты, экспорт.

### Обучающийся
- **«Назначено мне»:** список назначенных тем/уроков с дедлайнами и статусом.
- **«Каталог тем»** (только свободный): выбрать тему самому.
- **Экран сессии:** существующий FeynMap (чат слева + живой радар справа) — теперь заземлён в БЗ.
- **«Мои результаты»:** история сессий, итоговые карты, пробелы и рекомендации.

## 7. API (расширение к базовым /chat /assess /report)

- **Auth:** `POST /auth/login`, `GET /auth/me`, `POST /auth/logout`.
- **БЗ:** `POST /kb/documents` (upload), `GET /kb/documents`, `GET /kb/documents/:id`, `DELETE /kb/documents/:id`, `POST /kb/documents/:id/reindex`.
- **Темы:** `POST /topics`, `GET /topics`, `GET /topics/:id`, `PATCH /topics/:id`, `DELETE /topics/:id`, `POST /topics/suggest` (ИИ-черновик тем+компетенций из документа).
- **Компетенции:** `GET/POST/PATCH /topics/:id/competencies`.
- **Уроки:** CRUD `/lessons`.
- **Назначения:** `POST /assignments`, `GET /assignments?assignee=me|user_id|team`, `PATCH /assignments/:id` (статус).
- **Сессии:** `POST /sessions` (старт по `topic_id`/`assignment_id`), затем `/chat`, `/assess`, `/report` (с `session_id`); `GET /sessions/:id`, `GET /sessions?learner=`.
- **Аналитика:** `GET /analytics/learner/:id`, `GET /analytics/team`, `GET /analytics/org`.
- **Админ / пользователи:** `GET /users` (поиск/фильтр/пагинация), `POST /users` (создать + пригласить), `POST /users/import` (CSV/XLSX), `PATCH /users/:id` (правка/смена роли), `POST /users/:id/deactivate`, `POST /users/:id/reset-password`, `GET /roles`, `GET /org/settings`, `PATCH /org/settings`.

Каждый эндпоинт защищён RBAC-middleware (проверка permission + scope: своё/команда/организация).

## 8. Порядок сборки (по фазам)

1. **БД + авторизация + RBAC + админ-панель.** Postgres + pgvector, схема, JWT-логин, роли/права, seed 6 ролей и тестовых пользователей, middleware проверки прав. **Админ-панель: заведение пользователей (вручную + импорт CSV), смена роли, блокировка, сброс пароля.**
2. **Загрузка и индексация БЗ.** Upload → парсинг → чанкинг → эмбеддинги → pgvector; статусы и прогресс в UI Админа.
3. **Темы / уроки / компетенции.** Конструктор + ИИ-черновик из материала; утверждение и публикация; сбор уроков.
4. **Назначения.** Консоль назначений + «Назначено мне»; статусы и дедлайны.
5. **RAG в ядро.** Заземление Студента/Ассессора на БЗ; `evidence` и `source_refs` ссылаются на источник; gap при противоречии эталону.
6. **Аналитика и отчёты.** Карты по команде/организации, индивидуальные отчёты, экспорт; экран ревью/override для Ассессора.
7. **Полировка ролевых интерфейсов.** Скрытие недоступного, пустые состояния, онбординг под каждую роль.

## 9. Критерии готовности

- [ ] Админ в админ-панели заводит пользователя (вручную или импортом CSV), назначает роль и привязку — пользователь получает доступ.
- [ ] Админ загружает PDF (PMBoK/учебник) → документ индексируется, виден статус.
- [ ] Назначающий создаёт тему из документа (с ИИ-черновиком компетенций) и публикует; собирает урок.
- [ ] HR/наставник назначает тему/урок обучающемуся с дедлайном; тот видит её в «Назначено мне».
- [ ] Свободный обучающийся выбирает тему из каталога сам; назначаемый — нет (право заблокировано).
- [ ] В сессии вопросы Студента и оценка Ассессора **опираются на загруженный материал** (evidence ссылается на источник).
- [ ] Ассессор может посмотреть и скорректировать авто-оценку.
- [ ] Аналитика показывает карту компетенций по человеку и по команде/организации.
- [ ] Все права проверяются на бэке (не только в UI).

## 10. Безопасность и нюансы

- API-ключи (Anthropic, OpenAI) — **только на бэке**.
- RBAC — на сервере; UI-скрытие не считается защитой. Проверять и permission, и scope (свой/команда/орг).
- Валидация загружаемых файлов (тип, размер, антивирус-хук при возможности); БЗ может содержать конфиденциальные документы — изоляция по `org_id` на уровне всех запросов.
- Персональные данные обучающихся и результаты оценки — доступ строго по роли/scope; вести `AuditLog`.
- ИИ-оценка — вспомогательный инструмент: для ассессмента с последствиями для сотрудника предусмотреть ревью Ассессором (human-in-the-loop).
